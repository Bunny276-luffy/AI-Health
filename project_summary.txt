========================================================================
NeuroScan: Uncertainty-Aware AI Tumor Detection System
Comprehensive Project History and Implementation Details
========================================================================

1. INITIAL PROJECT SCOPE & SETUP
--------------------------------
Goal: Build a full-stack, cross-platform web application for advanced AI medical imaging, capable of:
- Tumor detection via uploaded scans (MCI, CT, Ultrasound).
- Calculating Epistemic (model ignorance) and Aleatoric (data noise) uncertainties.
- Longitudinal progression analysis (comparing baseline vs. current scans).
- Generating PDF diagnostic reports.

Architecture:
- Frontend: Next.js 14, React, Tailwind CSS (Dark Medical Theme)
- Backend: Python FastAPI, OpenCV, Uvicorn

2. PHASE 1: THE MOCK IMPLEMENTATION (Foundation)
------------------------------------------------
To establish the complex frontend/backend data flow, we first built the system using a "Mock" inference pipeline.
- Backend routing established for `/analyze`, `/progression`, and `/report`.
- Frontend Upload Panel created supporting drag-and-drop.
- Frontend Visualization Panel built using HTML Canvas to display uploaded scans, overlay heatmap dots, and draw tumor bounding boxes based on the mock backend JSON response.
- Uncertainty Panels built to dynamically render confidence vs. uncertainty metrics using progress bars.
- ReportLab integrated into the backend to dynamically generate a PDF report containing the mock JSON inference results and stream it back to the frontend for download.
- Fixed React Hydration errors caused by browser extensions injecting code into the HTML document.
- Fixed nested `.git` folder issues preventing the `frontend` folder from tracking correctly on GitHub.

3. PHASE 2: REAL DEEP LEARNING INTEGRATION (Current Step)
---------------------------------------------------------
Following the successful mock build, we transitioned to a real mathematical deep learning pipeline.

Step 1: Deep Learning Integration (COMPLETED)
- Analyzed PyTorch Hub to find an appropriate medical model: Mateusz Buda's Brain Segmentation PyTorch U-Net.
- Wrote robust downloader scripts (`robust_downloader.py`, `export_unet_local.py`) to bypass PyTorch Hub network timeouts by cloning the repository locally.
- Exported the PyTorch U-Net computational graph into `unet.onnx` so it can be run via ONNXRuntime, a highly optimized CPU/Edge inference engine.
- Rewrote `backend/models/inference.py`:
    - Reads binary image data into OpenCV.
    - Mathematically standardizes the image using ImageNet means `[0.485, 0.456, 0.406]` and scales to `256x256`.
    - Feeds the `[1, 3, 256, 256]` tensor into the ONNX session.
    - Applies a Sigmoid activation to the logit output to get the real probability map.
    - Generates authentic bounding boxes and heatmap coordinates by clustering the high-activation (`> 0.5`) regions from the U-Net probability map.

4. NEXT STEPS REMAINING (The 5-Step Plan)
-----------------------------------------
Step 2: Real Uncertainty Implementation (MC Dropout)
- We will replace the static uncertainty variables with Monte Carlo (MC) Dropout.
- The ONNX model will execute 10 times per image.
- We will calculate the inter-prediction mathematical variance to generate authentic Epistemic and Aleatoric Uncertainty metric values.

Step 3: Self-Calibrating Logic
- Integration of `SimpleITK`.
- If Epistemic Uncertainty exceeds `0.3`, we will execute image registration algorithms to stabilize the prediction automatically.

Step 4: DICOM & Metadata
- Integration of `pydicom` to read `.dcm` files natively.
- Extract "Patient Age", "Modality", and "Slice Thickness" dynamically.
- Embed this new clinical metadata into the PDF generated by ReportLab.

Step 5: Frontend Vibe-Coding
- Visualizing the authentic MC Dropout uncertainty maps as a toggleable overlay.
- Display conditional UI "Clinical Warnings" only when the true computed Epistemic metric passes the 0.3 threshold.

Deployment Status: 
- Frontend is configured with dynamic environment variables (`NEXT_PUBLIC_API_URL`) and `vercel.json` for serverless deployment on Vercel.
- Backend is restricted to containerized environments (Docker, Render, Railway) due to the large memory requirements of ONNXRuntime and the U-Net weights file.
